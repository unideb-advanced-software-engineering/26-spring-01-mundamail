---
title: 'ADR-003: API Gateway as the Single Security Enforcement Point'
description: The descision on security architecture to be implimented
---

- Status: Active

---

## In the context of
- the Government Messaging System's perimeter layer, which must authenticate all inbound traffic, enforce transport security, and prevent malicious content from entering the message pipeline
- a system where both government agency REST clients and citizen-facing web/mobile interfaces share the same entry path into the modular monolith

---

## facing the need for
- OAuth 2.0 token validation for all API consumers before any internal module processes a request (ASR-006)
- TLS 1.2+ enforcement for all data in transit across every client type, with centralised certificate management (ASR-007)
- blocking of executable attachments (.exe, .bat, .sh) at the earliest possible point in the pipeline, before they reach the messaging or storage layers (ASR-016)
- a single, auditable enforcement surface to support ISO/IEC 27001 compliance (SE-MM-007)
- an account lockout and admin notification mechanism after 5 failed login attempts (SE-MM-006)

---

## we decided for
- an API gateway positioned as the sole entry point for all inbound traffic, executing three ordered checks before routing any request to the modular monolith:
  1. TLS 1.2+ termination and cipher suite enforcement (ASR-007)
  2. OAuth 2.0 Bearer token validation against the Auth module's identity server (ASR-006)
  3. Multipart content inspection for blocked attachment extensions (ASR-016)
- requests failing any check are rejected immediately with the appropriate HTTP status code (401 for auth failure, 422 for blocked content) and a 'msg.blocked' or 'auth.failed' event is published to the dead letter queue
- internal module communication occurs over a trusted internal network without per-hop TLS, reducing latency on the hot path

---

## achieving
- a single, provably auditable enforcement point for all three security ASRs — no internal module can receive an unauthenticated, unencrypted, or content-unsafe request (ASR-006, ASR-007, ASR-016)
- elimination of the distributed enforcement surface that would arise if each module independently validated tokens or inspected content, reducing the attack surface and simplifying ISO/IEC 27001 audit evidence (SE-MM-007)
- ordered short-circuit evaluation — TLS is verified first (cheapest), then token validation (moderate cost), then content inspection (most expensive) — so no expensive work is done on unauthenticated or unencrypted requests
- reduced internal latency, since TLS termination at the gateway means modules communicate without cryptographic overhead on every in-process call
- full visibility into rejected traffic via DLQ events, giving administrators a complete record of blocked and failed requests without those events polluting the main audit log (ASR-010)

---

## accepting that (we will have)
- the gateway as a potential single point of failure for all inbound traffic, requiring it to be deployed with redundancy and its own high-availability configuration consistent with the 99.9% uptime requirement (ASR-001)
- a risk of the gateway becoming a performance bottleneck under extreme load if content inspection is not implemented efficiently — attachment inspection in particular must be bounded by file size limits (P-MM-006: 25MB maximum)
- the need to maintain and rotate TLS certificates centrally, which introduces a certificate management operational burden
- internal network traffic between the gateway and monolith assumed to be trusted — any breach of that internal network segment bypasses TLS enforcement, requiring appropriate network-level controls as a compensating measure
