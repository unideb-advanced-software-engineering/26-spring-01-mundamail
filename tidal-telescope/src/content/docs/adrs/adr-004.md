---
title: 'ADR-004: Centralised In-Process RBAC'
description: The descision on implementation of role based access to system modules
---
# 

- Status: Active

---

## In the context of
- the Government Messaging System's authorisation model, which must enforce role-based access control consistently across all modules — Auth, Messaging, API bridge, Admin, and attachment handling
- a system where citizens, government agency API clients, and administrators have distinct, non-overlapping permission sets that must be enforced at every operation boundary

---

## facing the need for
- system-wide RBAC where users access only their own data and administrators are limited to authorised actions, with no module able to silently bypass the policy (ASR-009, SE-MM-004)
- a centralised authorisation module that all components consult, rather than distributed, per-service policy enforcement (ASR-013)
- RBAC enforcement at the command boundary in the CQRS pattern — write operations must be authorised before any state change is committed
- audit logging of all authorisation decisions as part of the 12-month tamper-evident audit trail (ASR-010)
- compliance with ISO/IEC 27001, which requires demonstrable, consistent access control across the entire system (SE-MM-007)

---

## we decided for
- a dedicated RBAC module within the modular monolith, called in-process by every other module before any privileged operation is executed
- three primary roles: Citizen (read/write own messages only), Agency (send via API, read delivery status), Administrator (create/modify/deactivate accounts, approve mass messaging, access audit logs)
- all authorisation decisions recorded as events on the Kafka audit topic, making the access control history part of the immutable audit log
- the RBAC module holding no mutable state of its own — it reads role assignments from the shared persistence layer and evaluates policies at call time

---

## achieving
- a single, in-process enforcement surface for all authorisation decisions — no module can skip the RBAC check via a network misconfiguration or missing middleware, as the call is a direct in-process invocation (ASR-009)
- consistent policy application across all entry paths (citizen UI, agency API, admin portal) since all paths converge on the same module instance within the monolith
- straightforward auditability — every authorisation decision is an event in the Kafka log, satisfying the 12-month audit retention requirement and providing evidence for ISO/IEC 27001 audits (ASR-010, SE-MM-007)
- simplified cross-module refactoring — because RBAC is in-process, role policy changes are a single-module update with immediate effect, rather than a coordinated deployment across independent services
- alignment with the CQRS pattern — the command boundary is the natural authorisation checkpoint, and the in-process RBAC module is invoked there without a network round-trip

---

## accepting that (we will have)
- the RBAC module becoming a shared dependency of all other modules within the monolith — changes to its interface require coordinated updates across all callers, increasing the cost of role model evolution
- no independent scaling of the authorisation logic — if RBAC evaluation becomes a CPU bottleneck under extreme load, it cannot be scaled independently without extracting it as a separate service
- a requirement for disciplined testing of the RBAC module, as a regression in its logic has system-wide security implications rather than being isolated to a single service
- role assignment data residing in the shared persistence layer, creating a coupling between the RBAC module and the database schema that must be managed carefully during schema migrations
