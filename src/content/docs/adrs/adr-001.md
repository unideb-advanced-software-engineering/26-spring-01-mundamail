---
title: 'ADR-001: Hybrid Architecture for the Government Messaging System'
description: The architectural decision for the chosen architecture
---

# 

- Status: Active

---

## In the context of
- the Government Messaging System, a secure messaging platform enabling government agencies to communicate with citizens via a public REST API and a web/mobile user interface
- the subsystem responsible for accepting, routing, delivering, logging, and archiving messages across both agency-initiated and citizen-initiated flows

---

## facing the need for
- delivery of messages to potentially 50,000 recipients within 10 minutes, via asynchronous fan-out, without blocking the caller (ASR-004)
- an API response time of ≤2 seconds for 95% of calls under concurrent load (ASR-003)
- support for 10,000 concurrent users without degradation, requiring horizontal scalability (ASR-005)
- system-wide RBAC enforcement and OAuth 2.0 authentication that cannot be bypassed by any component (ASR-006, ASR-009)
- a tamper-evident, append-only audit log of all logins, API calls, and admin actions retained for 12 months, and message logs retained for 5 years (ASR-010, ASR-015)
- a modular architecture where components are independently updatable without full redeployment (ASR-013)
- a REST-to-IMAP/SMTP bridge as a core architectural boundary between modern clients and legacy email protocols (ASR-011)
- 99.9% uptime with an RTO of 4 hours and RPO of 1 hour (ASR-001, ASR-002)
- integration with government identity systems via LDAP/SAML (ASR-012)

---

## we decided for
- a Hybrid Architecture consisting of:
  - a modular monolith for core domain logic, security enforcement, persistence, and web presentation
  - an Event-Driven Architecture (EDA) via Kafka for all asynchronous message delivery, audit logging, and archival flows

---

## achieving
- decoupling of API response time from message delivery time, allowing the API bridge to publish a `message.send.requested` event and return 202 immediately, satisfying the ≤2s SLA (ASR-003)
- parallel fan-out across Kafka partitions enabling 50,000-recipient mass messaging within 10 minutes without synchronous delivery bottlenecks (ASR-004)
- horizontal scalability of the stateless monolith behind a load balancer, with session state externalised to Redis, absorbing 10,000 concurrent users without thread starvation (ASR-005)
- centralised, in-process enforcement of OAuth 2.0 and RBAC within the monolith, eliminating the distributed enforcement surface that would exist in a microservices design (ASR-006, ASR-009)
- an immutable Kafka event log that naturally serves as the tamper-evident audit trail, with retention policy and cold-tier archival satisfying 12-month and 5-year compliance requirements (ASR-010, ASR-015)
- clean module boundaries enforced through code-level contracts rather than network contracts, enabling independent updates without the operational overhead of a microservices mesh (ASR-013)
- a simpler disaster recovery posture than microservices — restoring a single deployable unit and replaying Kafka offsets achieves the 4-hour RTO without orchestrating recovery across independent services (ASR-001, ASR-002)

---

## accepting that (we will have)
- higher infrastructure complexity compared to a purely synchronous monolith, due to the need to operate, monitor, and tune a Kafka cluster
- more difficult end-to-end testing and debugging caused by asynchronous delivery flows, requiring investment in distributed tracing and consumer offset monitoring
- no inherent guarantee of delivery order across partitions, requiring the application layer to handle message sequencing explicitly where ordering matters
- a scalability ceiling if individual modules require independent scaling at significantly different rates — at that point, extracting a module as a separate service would need to be revisited
